## 4. 结果对比与分析

### 4.1 功能正确性

- `v2`、`v3` 单元测试均通过，说明当前实现功能可用，边界与并发场景未触发明显错误。

### 4.2 性能表现概览

- `v1`：内存池版本明显慢于系统分配（108ms vs 3ms），说明该基础实现在当前测试规模下开销较大。
- `v2`：表现最好，三类场景中有两类优于 `new/delete`（多线程、混合大小）。
- `v3`：在本次数据中整体落后于 `new/delete`，尤其混合大小场景差距较大。

### 4.3 可能原因（结合当前数据）

- 测试模型差异较大（`v2` 与 `v3` 的分配规模和分布并不完全一致），不能只看绝对毫秒值做跨版本结论。
- 内存池在某些场景下会引入额外管理成本（链表维护、批量归还、竞争同步），当分配器本身很快时可能不占优。
- WSL 下的调度、内存映射和系统分配器实现，也会影响微基准结果波动。

## 5. 本次排障记录

- 在 `v2`、`v3` 编译过程中出现 `malloc/free was not declared`。
- 根因：对应 `ThreadCache.cpp` 缺少头文件 `<cstdlib>`。
- 处理后重新构建通过，`unit_test` 与 `perf_test` 均可执行。

## 6. 结论

- 本地环境与运行链路已打通，三个版本都能在 WSL 下完成编译与运行。
- 从本次单轮数据看：`v2` 的性能形态最好；`v1` 和 `v3` 在当前测试下没有体现优势。
- 建议后续做 5~10 轮重复测试并取均值/方差，再用于版本优劣结论或简历材料。

## 7. 建议的后续实验

- 固定随机种子，减少波动（便于复现）。
- 统一三版本测试规模（线程数、分配次数、大小分布）后再横向比较。
- 分别测试 `Debug` 与 `Release`，并记录 CPU 占用、峰值内存。
- 增加高并发小对象场景（通常更能体现内存池收益）。
